# real_measures — committed harnesses behind the cross-check figures

The parent README quotes real-tokenizer and latency figures; these scripts
are how they were produced, so every number stays reproducible.

| Script | What it measures | Prereqs |
|---|---|---|
| `measure.mjs` | Real cl100k tokens (gpt-tokenizer) of the benchmark corpus, raw vs compiled, per budget — verifies the output fits the budget in *real* tokens and that the Node stack is byte-deterministic | `cd crates/velesdb-node && npm ci && npm run build && npm install --no-save gpt-tokenizer`, then `node measure.mjs` |
| `stress.mjs` | Compile latency and peak RSS at the DoS caps (1024×1 KB, 10 MB, 64 MB, 1 MB repetitive log, 1024-duplicate avalanche) on the release addon | same build, then `node stress.mjs` |
| `mcp_e2e.py` | Drives the real MCP server over stdio JSON-RPC and exercises all four context tools end-to-end (also usable as a latency probe) | `cargo build --release -p velesdb-memory`, then `python3 mcp_e2e.py` from the repo root |
| `tri_engine_rescue.mjs` | **The tri-engine benchmark**: an incident knowledge base stored per the velesdb-memory skill (cause/fix chains via `remember` + `relate`), then per question an A/B/C over the same store — vector-only recall vs fused `memory_scope` (HNSW seed + graph BFS + fusion) at default and raised `graph_boost`. Scores which answer-bearing facts each mode surfaces, with real cl100k token counts. Ends with the **RL×graph synergy case** (EPIC-P-071/US-002): a fact reinforced via `feedback` and reachable only through the typed-edge walk out-ranks a merely-similar fact once `policy.importance` is active, with the four-signal reason ventilation printed | same build as `measure.mjs`, then `node tri_engine_rescue.mjs` |
| `agent_session.mjs` | **The agent-conditions benchmark**: a deterministic 12-turn coding session whose context accumulates (turn history, the same file read twice, a 180-line CI log, constraints) — compiled before every "LLM call" the way the `velesdb-context-optimizer` skill prescribes. Counts everything with real cl100k tokens; also reports the stateless-compile latency and the byte-stability of the cache-marked prefix | same build as `measure.mjs`, then `node agent_session.mjs [budget]` |
| `exact_estimator.mjs` | **Exact-estimator gap** (EPIC-P-071/US-005): per content class (English/French prose, repetitive logs, Rust code, URLs, Markdown, JSON, digit-dense ids/dates, CJK) compiles a single fragment with a huge budget so `insights.tokens_in` is exactly the default `HeuristicEstimator`'s estimate, and compares it against the real cl100k count of the same bytes — reproduces the calibration margins documented on `HeuristicEstimator` (always a safe over-count) on fresh corpus snippets. Measured (two runs, identical): English +63.8%, French +52.5%, repetitive logs +52.4%, Rust code +30.6%, digit-dense ids/dates +30.9%, Markdown +13.0%, JSON +13.6%, URLs +11.8%, CJK +9.6% — all safe over-counts, none under | same build as `measure.mjs`, then `node exact_estimator.mjs` |
| `cache_prefix.mjs` | **Cache-prefix stability** (EPIC-P-071/US-008): 10 consecutive compiles where cache-marked fragments never change but the volatile log content does every turn; measures the byte-stable common prefix between every consecutive pair against the cache section's own length, plus a cost framing (an injected EXAMPLE `policy.pricing` table, never a hardcoded rate) of the naive full-input-rate cost of resending that prefix instead of caching it. Measured (two runs, identical): the cache section is a byte-stable 100% prefix across all 9 consecutive turn pairs | same build as `measure.mjs`, then `node cache_prefix.mjs` |

Token figures are machine-independent; latency and RSS vary with hardware
(reference figures in the parent README were measured on Apple Silicon,
release profiles).
